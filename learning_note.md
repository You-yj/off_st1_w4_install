# 一、任务理解

## 1. 什么是结果责任

结果责任指开发人员需要对自己编写代码的**运行正确性、边界兼容性、异常容错性**全权负责，不能仅保证代码无语法报错、能运行，还要保证：合法输入输出符合数学逻辑、边界空数组不会崩溃、非法输入会主动抛出规范异常；后续迭代修改代码时，原有功能不会被破坏。

简单来说：写出代码只是完成基础工作，**保证代码在全场景下可靠运行**才是开发需要承担的结果责任。

## 2. 为什么业务代码必须配套测试代码

1. **验证逻辑正确性**：手动测试覆盖场景有限，单元测试可以批量校验常规输入、边界空数组、非法形状等场景，确保算子 / 业务计算结果和理论预期完全一致
2. **迭代防退化**：后续修改底层算子（如 FEALPy 的 bm.tril、bm.dot、矩阵加法）时，一键运行所有测试，快速发现改动破坏原有功能，避免线上隐性bug
3. **规范异常容错**：测试覆盖非法参数、维度不匹配场景，强制校验代码是否主动抛出对应异常，防止非法输入导致程序静默出错、输出错误数值
4. **降低排查成本**：测试用例失败会精准定位出错输入、后端、场景，相比手动复现问题效率大幅提升
5. **交付可信代码**：代码能跑仅代表无语法错误，配套全量测试才能证明逻辑可信，是开发交付的基础标准

# 二、知识总结

## 1. 参数化测试 `@pytest.mark.parametrize`

1. 核心作用：消除重复测试样板代码，一组逻辑批量覆盖多组输入输出，结构更加清晰、维护成本更低
2. 使用场景：矩阵 tril 提取、矩阵加法、向量点积等大量同结构正向计算用例。
3. 核心优势：新增测试场景仅需在数据集列表新增一条元组，不用复制整个测试函数；每组参数独立执行，单组失败不影响其余用例，报错精准定位输入样例。

## 2. 异常捕获校验 pytest.raises ()

1. 核心作用：编写反向 / 异常测试用例，校验代码对非法输入的容错能力，断言代码会抛出指定类型异常。
2. 使用场景：矩阵加法形状不匹配、dot 向量长度不一致、tril 传入 k=None 非法参数。
3. 关键细节：numpy 与 pytorch 后端维度错误抛出异常类型不同（ValueError / RuntimeError），可传入元组同时捕获多种异常；必须将**实际运算逻辑**放入 with 代码块，不能仅做布尔判断。

4. `with pytest.raises(ValueError)` 的作用

   这段代码的含义：

   > 我确定这行代码所包含的程序一定会抛出 ValueError，如果真抛出来了，测试通过；如果没抛异常，测试失败。

5. `with pytest.raises((ValueError, RuntimeError))` 的作用

   这段代码的含义：

   > 我确定这行代码所包含的程序一定会抛出 ValueError 或者  RuntimeError，如果真抛出来了，测试通过；如果没抛异常，测试失败。

## 3. 自定义测试标记 markers + pytest.ini

1. 核心作用：对测试用例分类，实现批量筛选执行，替代大量 skip 标记。

2. 使用流程：项目根目录创建 pytest.ini 注册标记（basic/abnormal/empty）；函数添加 @pytest.mark.xxx；使用`pytest -m 标记名`筛选运行。

3. 实用命令：

   ```python
   pytest test.py -m basic        # 仅执行基础正向用例
   pytest test.py -m "not abnormal" # 跳过异常测试
   ```

4. 价值：开发调试时仅跑基础快速用例提速，完整验收时执行全部用例，分层管理测试流程。

# 三、实践复盘

## 坑 1：双层参数化装饰器顺序错误，报 fixture 'backend' not found

- 问题现象：

  内层写 backend 参数化、外层写矩阵样例参数化，函数入参顺序不匹配，pytest 将参数变量识别为内置夹具，直接报错。

- 原因

  pytest 读取装饰器从上至下解析，参数顺序必须和函数形参从左到右严格对应。

- 解决方案

  外层装饰器写 backend 后端参数，内层写测试数据集；函数参数顺序与装饰器顺序一一对应。

```python
# 正确顺序
@pytest.mark.parametrize("backend", ["numpy", "pytorch"])
@pytest.mark.parametrize("raw_a, raw_b, exp", data_test_add)
def test_add(backend, raw_a, raw_b, exp):
```

## 坑 2：异常测试逻辑写错，只判断形状不执行运算，用例无意义

- 问题现象

  测试 矩阵相加 与 dot 维度不匹配异常时，只写`bm.shape(a) != bm.shape(b)`，没有执行加法 与`bm.dot()`，且bm没有shape这个函数。

  后取消设置后端，均为默认numpy后端，只写`np.shape(a) != np.shape(b)`，with 块内无报错，测试虚假通过，无法校验算子容错。

- 原因

  布尔比较语句不会抛出任何异常，pytest.raises 捕获不到错误，用例形同虚设

- 解决方案

  with 代码块中**必须执行待测试的算子运算**，依靠底层运算触发异常

```python
with pytest.raises(ValueError):
        _ = a + b
```

```python
with pytest.raises(ValueError):                                                    
        _ = bm.dot(vec1, vec2) 
```

## 坑 3：异常类型写死 TypeError，切换 pytorch 后端测试失败

- 问题现象

  维度不匹配场景固定捕获 TypeError，numpy 抛 ValueError、torch 抛 RuntimeError，两端后端测试均失败。

- 原因

  维度不兼容属于数值运算错误，不是参数类型错误；两套后端内置异常类型不统一。

- 解决方案

  传入元组同时捕获两种异常：

```python
with pytest.raises((ValueError, RuntimeError)):
    _ = bm.dot(vec1, vec2)
```

## 坑 4：测试代码能运行通过，但逻辑完全错位（ dot 异常测试后面却是加法运算）

- 问题现象

  test_dot_abnormal 函数内部执行`vec1 + vec2`加法，而非 bm.dot，测试通过但没有校验点积接口，代码跑通但结果不可信。

- 核心感悟：

  代码无报错、测试显示 passed ≠ 功能符合预期。仅语法和异常匹配校验通过，不代表测试逻辑正确，需要人工核对测试运算与函数主题是否对应。

- 解决方案

  异常用例运算逻辑与函数名称保持统一，dot 异常只调用 bm.dot，加法异常只执行矩阵相加。

# 四、学习收获

1. 建立完整单元测试思维：明白 “代码能运行” 只是最低标准，结果责任要求覆盖**常规场景、边界空数组、非法异常**三层测试，单一手动运行无法保证代码可靠。
2. 熟练掌握 pytest 工业化测试写法：会使用参数化批量管理多后端、多输入样例；会通过 raises 校验异常容错；能用自定义标记分类筛选测试，适配 FEALPy 多后端算子测试场景。
4. 形成问题排查思路：遇到 fixture 报错先核对装饰器顺序；异常捕获失败先核对运算逻辑、异常类型；张量报错区分 numpy/pytorch 后端差异。
5. 深刻理解「代码跑通不等于结果可信」：本次 dot 异常测试写错运算逻辑但测试依然 passed 的经历证明，无校验逻辑的测试没有价值，编写用例时必须对齐测试目标，不能只追求无报错。
6. 规范工程化测试习惯：正向计算用参数化统一管理，**异常用例单独拆分**；通过配置文件注册标记分层执行，适配大型数值计算库（FEALPy）的批量单元测试开发。
